# 用户洞察看板原型交付开发审计报告（业务流程、功能口径与可行性）

本审计报告专为**原型交付开发阅读**而设计。我们剥离了对静态代码美观度、样式层面的审计，聚焦于开发团队最关心的**业务流程闭环性**、**口径与指标定义的清晰度（SQL逻辑指引）**以及**数据跨系统整合的技术可行性**。

---

## 1. 业务流程与数据流向闭环审计

在将原型转化为真实系统的开发过程中，需重点保障以下两个核心流程在前端和后端的闭环。

### 1.1 全局筛选传参流向（多维联动流程）
* **现状描述**：原型中三个 Tab（渠道效果、培育运营、用户洞察）都拥有独立的筛选栏。
* **开发实现要求**：
  * **状态共享 vs. 状态隔离**：在实际业务操作中，建议将筛选器的状态在三大 Tab 间进行**共享**（即切换 Tab 时，已选择的“时间范围”、“车系”、“门店”等条件应保持一致，避免用户在切换页签时重复选择）。
  * **下钻弹窗传参继承（核心流程）**：
    当用户点击“城市投放效果”或“渠道线索质量”柱状图下钻时（例如，点击“广州市”查看专营店意向分布弹窗）：
    ```mermaid
    graph TD
        Filter[全局筛选栏: 车系/渠道/落地平台/时间维度...] -->|1. 收集过滤条件| SQL[后端数据接口]
        Chart[主看板图表] -->|2. 点击特定区域/广州市| Drill[下钻弹窗/专营店分布]
        Filter -->|3. 继承过滤条件| Drill
        Drill -->|4. 带上全局条件 + 广州市 重新请求| SQL
    ```
    *开发人员应确保下钻弹窗的数据接口在执行 SQL 时，必须将**主看板筛选栏中的所有过滤条件**（车系、渠道、落地平台、时间维度等）作为 Query 参数传入，不可只按城市或大区单一维度过滤。*

### 1.2 业务实体流转流程（从线索到画像）
本系统涉及三个不同阶段的数据模型，数据流向链条如下：
1. **留资阶段 (留资记录表 - ADS3)**：用户在官网、车展等多渠道、多媒体发生的多次留资行为。
2. **培育阶段 (线索跟进表 - ADS1)**：总部电销坐席接收到留资并合并为唯一线索后，进行外呼、清洗，并分发/下发到专营店。
3. **用户画像阶段 (去重用户画像表 - ADS2)**：根据手机号对线索进行去重，融合通话历史与标签，沉淀至基础画像库，用于宏观洞察及工作台 (`workbench.html`) 预测画像展示。

## 2. 核心功能说明与 SQL 取值逻辑审计（消除二义性）

为了避免开发团队因原型文案模糊而产生开发偏差，以下对核心业务口径给出明确的数学与逻辑定义。

### 2.1 10级线索等级的判定与 COALESCE 优先级
在 [business-rules.md](file:///Users/huhaowen/Documents/35-用户洞察/user_insight_pm%20v1.0/memory/business-rules.md#L40) 中，线索等级被定义为 10 级，但在数仓表结构中，同一条线索可能存在多个节点产生的级别记录。

* **口径规则**：级别取值优先级为：`下发级别` > `跟进级别` > `初始级别`。
* **SQL 逻辑参考**：
  ```sql
  -- 在 ADS1 层级，生成线索最终级别的计算逻辑
  SELECT 
      lead_id,
      COALESCE(下发级别, 跟进级别, 初始级别) AS final_lead_level
  FROM ads1_lead_detail;
  ```
  *说明：若线索已分发给门店，以门店收到的 `下发级别` 为准；若尚未分发但总部坐席已跟进，以 `跟进级别` 为准；若坐席还未联系，以线索生成时的 `初始级别` 为准。*

### 2.2 培育跟进过程“双口径”混合计算公式
[培育跟进过程卡片](file:///Users/huhaowen/Documents/35-用户洞察/user_insight_pm%20v1.0/memory/business-rules.md#L39-L45)同时展示了线索维度和通话维度的指标，开发必须理清它们的分母与分子关系：

| 指标名称 | 口径维度 | 分子定义 | 分母定义 | 计算公式 |
| :--- | :--- | :--- | :--- | :--- |
| **呼叫线索数** | 线索维度 | 发生过至少 1 次呼叫的去重线索数 | - | `COUNT(DISTINCT CASE WHEN 外呼次数 > 0 THEN 线索编码 END)` |
| **接通线索数** | 线索维度 | 发生过至少 1 次接通的去重线索数 | - | `COUNT(DISTINCT CASE WHEN 接通次数 > 0 THEN 线索编码 END)` |
| **单线索平均呼叫** | 通话/线索 | 总呼叫次数 | 呼叫线索数 | `SUM(外呼次数) / 呼叫线索数` |
| **通话接通率** | 通话维度 | 总接通次数 | 总呼叫次数 | `SUM(接通次数) / SUM(外呼次数)` |
| **线索接通率** | 线索维度 | 接通线索数 | 呼叫线索数 | `接通线索数 / 呼叫线索数` |

*开发可行性校验：在设计数仓明细表时，需要一张“通话明细表（记录每一次呼叫的 duration 和状态）”和一张“线索主表”。上述指标均可通过在通话明细表上基于 `lead_id` 进行 `COUNT(DISTINCT)` 和 `COUNT()` 关联计算得出，数据可行性完全支持。*

### 2.3 桑基图 (Sankey) 留资流转算法
桑基图（触媒/触店习惯）用于反映线索的变迁轨迹。

#### A. 首次留资渠道 vs 末次留资渠道
对于一个手机号在 ADS3 中可能存在的多次留资流转记录：
* **首/末次识别算法**：
  ```sql
  WITH ordered_records AS (
      SELECT 
          phone_number,
          channel_name,
          create_time,
          ROW_NUMBER() OVER(PARTITION BY phone_number ORDER BY create_time ASC) as first_rank,
          ROW_NUMBER() OVER(PARTITION BY phone_number ORDER BY create_time DESC) as last_rank
      FROM ads3_touch_records
  )
  SELECT 
      a.phone_number,
      MAX(CASE WHEN first_rank = 1 THEN channel_name END) as first_touch_channel,
      MAX(CASE WHEN last_rank = 1 THEN channel_name END) as last_touch_channel
  FROM ordered_records
  GROUP BY a.phone_number;
  ```

#### B. 桑基流向长尾数据聚合算法（防前端崩溃）
* **规则描述**：由于末次媒体可能有数百个，直接渲染会导致 SVG 布局混乱。必须将第 50 名以后的媒体进行聚类收纳。
* **SQL/前端处理逻辑**：
  ```sql
  -- 计算首次媒体 R_source -> 末次媒体 R_target 的流向数并进行 Top 50 裁剪
  WITH target_ranks AS (
      SELECT 
          last_media,
          COUNT(1) as lead_count,
          ROW_NUMBER() OVER(ORDER BY COUNT(1) DESC) as media_rank
      FROM ads3_touch_records
      GROUP BY last_media
  )
  SELECT 
      first_media,
      CASE WHEN tr.media_rank > 50 THEN '其他媒体' ELSE tr.last_media END AS last_media_group,
      COUNT(1) AS link_leads
  FROM ads3_touch_records tr
  GROUP BY first_media, last_media_group;
  ```
  *开发可行性：前端使用已有的 SVG 分页引擎渲染前 50 行，超过 50 个节点的长尾媒体由后端直接返回为“其他媒体”节点，渲染开销可控。*

## 3. 技术可行性评估与数据接入审计

### 3.1 跨系统数据整合可行性（线索主键打通）
* **可行性风险**：
  在“门店虚拟号跟进过程”以及“通话时间段统计”中，外呼行为通常发生在**第三方呼叫中心系统**或**企微客服助手**；而线索分配与交车数据则存放在营销平台 **e3s** 中。
* **可行性通过前提**：
  开发团队必须确保在离线数据数仓（ODS -> DWD）层级，呼叫系统的**通话录音流水表**能通过 `线索编码 (lead_id)` 或 `手机号码 (phone_number)` 与 e3s 的**线索基础信息表**实现 100% 物理关联。若两者缺乏关联主键，则“虚拟号呼叫次数”和“接通率”将无法按大区、车系和渠道进行归因筛选。

### 3.2 门店树模糊搜索性能（591家门店）
* **现状描述**：原型中使用静态树形结构展示了全国 591 家门店，支持多选、省市区三级检索及文本搜索。
* **开发可行性建议**：
  * **前端渲染限制**：如果由前端一次性拉取并渲染 591 家门店的 DOM 树，且绑定多选交互，可能会在低端设备上发生轻微的响应延迟。
  * **推荐方案**：采用树形控件的“懒加载”技术（只在展开大区/省份时才去加载对应的专营店），或在搜索框输入超过 2 个字时触发异步后端接口检索，避免全量 DOM 的静态渲染。

## 4. 总结：开发交付合规性结论

经审计，**用户洞察看板原型（v1.0）在流程、口径和可行性上基本满足交付开发的标准。** 
原型文档中关于数据源 ADS1/ADS2/ADS3 的规划与计算公式定义清晰，后端数据架构师可直接将其映射至数仓的 DWD 宽表设计中。开发团队可依据本报告中提供的 SQL 示例和流程逻辑，进行数据模型的设计与编码。
